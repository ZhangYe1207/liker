# AI Chat 多会话持久化

## Problem Frame

Liker 的 AI 对话当前只存在于 `AIChatPanel` 的本地 state 里。关闭面板 / 页面刷新 / 浏览器重启，全部聊天记录连同推荐卡片一起消失。用户无法：
- 关闭面板后回看刚才的品味分析结果
- 隔天继续一条未完成的对话
- 把不同话题（品味分析 / 电影推荐 / 读书聊）分开维护

这是一个个人品味工具，AI 生成的内容（分析、推荐、观点）本身就是值得留存的资产。需要把对话从"临时 UI state"升级为"用户数据的一部分"。

## Requirements

- **R1**. 对话记录按用户持久化到 Supabase，关闭面板、页面刷新、退出浏览器后依然可见
- **R2**. 默认行为：打开 AI 面板时自动恢复用户「最近活跃」的那条对话；若用户从未有对话或全部已删除 → 显示欢迎页（等同现在的空态）
- **R3**. 标题栏显示当前对话标题；点击标题 → 展开下拉，列出该用户的全部对话（标题 + 最后更新时间）
- **R4**. 下拉列表中每条对话支持：点击切换 / 行内重命名 / 删除（带二次确认弹窗）
- **R5**. 「新建会话」按钮：清空面板、进入空态；新对话在用户发送第一条消息时才正式落库（懒创建）
- **R6**. 新对话标题默认为用户首条消息前 20 字；用户可随时手动重命名
- **R7**. Assistant 消息附带的推荐卡片（search_external 返回的电影/书/音乐）随消息一起持久化；重新打开旧对话时卡片完整渲染，「+ 收藏」按钮依然可用
- **R8**. 打开一条对话时按时间顺序加载并显示全部历史消息

## Success Criteria

- 发几条消息 → 关闭 AI 面板 → 重新打开 → 上次那条对话和全部消息原样出现
- 硬刷新浏览器（F5）→ 上次的对话仍然在
- 新建 3 条对话聊不同话题 → 下拉列表列出 3 条；点击切换后消息区切到对应对话，其他对话不受影响
- 对 AI 说「推荐一部悬疑电影」→ 看到推荐卡片 → 关面板 → 重新打开该对话 → 卡片还在，点「+ 收藏」能正常填入 AddEditModal
- 重命名一条对话「电影 2026 春」→ 刷新页面 → 下拉里仍是「电影 2026 春」
- 删除一条对话 → 确认 → 下拉列表立即移除；若被删的正是当前会话，自动切换到最近活跃的另一条（或空态）

## Scope Boundaries (非目标)

- 不做跨设备实时同步（打开时拉取一次，不订阅实时变更）
- 不做对话内搜索
- 不做置顶、归档、文件夹分组
- 不做消息级编辑 / 重新生成 / 删除单条
- 不做对话分享 / 导出
- 不做软删除 / 回收站
- 不做每个对话独立的 system prompt
- 不做上下文窗口管理优化（对话变长后全量发给 LLM，MVP 暂不裁剪；超长对话是后续独立话题）

## Key Decisions

- **多会话结构 over 单会话持久化**：用户明确要"按不同会话分开保存"，分主题管理比单条 localStorage 更符合实际使用模式
- **首条消息前 20 字做标题，不调 LLM 生成**：MiniMax 个人版 RPM 有限；额外的 LLM 调用成本没带来等比收益；首条消息已经足够代表对话主题；用户可重命名兜底精度问题
- **标题栏下拉 over 左侧 sidebar**：当前面板 420px 固定宽度，内嵌 sidebar 空间逼仄；下拉切换保持原布局、视觉噪音最小，代价是切换多一次点击（对低频操作可接受）
- **懒创建（R5）**：点「新建会话」不立即写库，第一条消息发出时才 INSERT，避免数据库里堆满空对话行
- **硬删除带二次确认**：个人工具、软删除过度工程；弹窗确认覆盖误操作风险
- **推荐卡片随消息持久化**：推荐本身是对话的高价值产出，重开对话看不到推荐 = 丢失上下文

## Dependencies / Assumptions

- Supabase RLS policies 按现有 items/profiles 模式延展（每条对话 `user_id` 字段 + 四条 CRUD policy）
- 假设单设备使用；多设备场景下「同时在两个设备编辑同一对话」的一致性问题延后处理
- 现有 `AIChatPanel` 组件主体保留；state 所有权会上移（具体形式在 plan 阶段定）

### Deferred to Planning

- [Affects R1][Technical] 流式输出过程中的持久化节奏：一条助手消息流完整了再写库，还是流一段存一段？前者实现简单，断网/崩溃时会丢；后者复杂但更健壮
- [Affects R2][Technical] 「最近活跃」怎么定义：对话的 `updated_at`（每条消息发送时 bump）vs 单独的 `last_opened_at`（用户切到这条时 bump）？两者在「看了但没回」的场景下语义不同
- [Affects R4][UX] 重命名交互：下拉行内点击标题即变输入框，还是弹出小 Modal？
- [Affects R7][Schema] 推荐卡片的存储方式：独立 `recommendations` 表（message_id 外键）vs 直接作为 assistant message 的 JSON 字段？前者规范、后者读写更快
- [Affects R8][Needs research] 多轮对话连续性：当前 `chat_with_rag`（backend/app/services/rag.py）只接收单条 user message（lines 82-97），不传历史。多轮要把历史消息一起塞到 messages 数组里。需要 planning 阶段确认并改造
- [Affects R6][Edge case] 用户发第一条消息前就点了「新建会话」里的 quick action（如「分析我的口味」）→ 这条系统预置文案要不要用来做标题？还是仍用用户真实首条？
